# Add an offline full-chain Factory system smoke

## Goal

Add one deterministic development/test-only journey that drives the manual Factory protocol through separate source-CLI processes against an isolated temporary Git repository and Factory store. The journey must prove triage, planning candidate/review/revision with planner-session continuity, local reviewed-plan materialization, implementation candidate/aggregate-review/revision with implementer-session continuity, exact reviewed-candidate promotion, implementation PR publication and merge acknowledgement, and terminal replay without repeated work.

The smoke must remain offline and outside Vitest, `pnpm test`, `pnpm test:watch`, pre-commit, and the ordinary `pnpm check` loop. `pnpm smoke:factory` is its explicit entry point; full `pnpm check:ci` runs it after the existing ordinary checks, while the existing plan-only GitHub Actions path continues to bypass `check:ci`. The accepted clarification supersedes the original plan-publication criterion: this item-file journey commits the locally approved plan as operator setup for implementation, but does not publish or acknowledge a plan PR.

## Changes

1. `scripts/smoke-factory.ts` — implement the single readable orchestration script using Node's type-stripped TypeScript and production CLI/configuration seams only.
   - Create an isolated `/tmp` fixture containing a Git repository, a local bare `origin`, an explicit Factory store/project id, an item JSON file, an ignored `.harness/` scratch area, and a feature branch based on `main`. Use a `file://localhost/tmp/<unique>.git` origin so the production publisher can validate a repository-shaped URL while every fetch/push remains local. Commit only generic fixture files and set deterministic Git identity/configuration.
   - Write fixture-local fake executables, not reusable fixture infrastructure: configure every Factory role's existing Codex `executable` field to one fake and prepend a minimal `gh` fake to the child `PATH`. Strip provider/tracker/GitHub credentials from child environments. The Codex fake must consume the SDK's real `codex exec --experimental-json` stdin/JSONL protocol, classify the stable triage/planner/spec-review/implementer/implementation-review/quality-review prompts, emit valid schema-shaped results and thread ids, and log each invocation. It writes only the requested planning draft or the fixture implementation file. On revision calls it must require `resume <original-thread-id>` before producing changed plan/code. The `gh` fake supports only `pr list` and `pr create`, persists one PR identity using the reviewed remote head, and logs calls; any unexpected command fails.
   - Run every station/continuation/publication/replay through a fresh `node bin/harness.ts ...` child with `--workspace`, `--item-file`, `--factory-store-root`, and `--factory-store-project-id`. Drive this exact journey: triage to `ready-to-plan`; planning candidate attempt 1; blocking spec review; `continue --decision revise`; candidate attempt 2 using the original planner session; passing review and local plan materialization; operator commit of that exact plan; implementation candidate attempt 1; aggregate implementation/quality review with one must-fix result; `continue --decision revise`; candidate attempt 2 using the original implementer session; passing aggregate review; implementation publication; a local no-fast-forward merge containing the reviewed head; merge acknowledgement; then a terminal implementation replay. Do not invoke planning publication commands.
   - Parse each command's JSON and assert the stable action handler, attempt, next reaction/wait reason, phase run continuity, and final `complete` outcome. Invoke read-only `factory inspect` in separate processes at phase and terminal boundaries to assert the reduced state, reaction, and artifact roots. Read the canonical lifecycle JSONL and assert the requested/candidate/review/continuation/publication/merge sequence, needs-changes then pass verdicts, candidate/review bindings, and implementation attempts `refs/harness/factory/<phase-run-id>/1` and `/2`. Assert attempt 1 is not promoted, the passing attempt-2 commit is simultaneously the reviewed event head, promoted feature-branch tip, published remote head, fake-PR head, and ancestor of the acknowledged merge.
   - Independently collect every artifact-ref-shaped object from lifecycle events, resolve it only beneath the roots reported by `factory inspect`, and verify its SHA-256. Snapshot first-round evidence before each revision and prove it remains byte-identical afterward. At each process boundary require the real index to match the current branch tree, except that a needs-changes implementation candidate may leave only its expected unstaged workspace edit; after passing promotion and through publication/replay require a clean worktree and index. Assert no plan PR event or external call exists, and fake invocation counts are exactly one triager, two planners, two spec reviewers, two implementers, and four aggregate reviewers. Before terminal replay, snapshot lifecycle bytes, all referenced evidence, Factory inspection state, Git refs/status, remote refs, fake PR state, and invocation logs; require replay to leave every snapshot unchanged.
   - Keep success output quiet apart from a compact pass/duration line and remove both workspace/store and bare-remote fixture roots. Track a named station around each setup/action/assertion; on failure print that station, bounded child stdout/stderr, and retained fixture/store/remote paths, then leave all evidence intact. Do not enforce elapsed time, but keep the deterministic fake journey simple enough to remain below the issue's roughly 20-second CI cutoff.

2. `package.json`, `Makefile` — add `pnpm smoke:factory` and a wrapped `make smoke-factory` target for the standalone source-CLI script. Keep `check` and `check-v` unchanged. Change `check-ci` to run `smoke-factory` only after its existing `check` prerequisite completes, preserving bounded gate diagnostics and avoiding a second workflow or Vitest project.

3. `test/docs-contracts.test.ts` — extend the existing command/testing contract seam to require the new package/Make command and testing-guide lane, and to pin the gate split: ordinary `check` excludes the Factory smoke, `check-ci` includes it, and `.github/workflows/test.yml` still sends plan-only pull requests through focused plan checks while other runs use `pnpm check:ci`. This is configuration coverage only; do not execute the system journey from Vitest.

4. `docs/contributing/testing.md`, `docs/contributing/script-command-surface.md` — document four distinct lanes and ownership: default unit/workflow/CLI Vitest coverage, built distribution smoke, explicit offline Factory system smoke, and separately authorized optional live verification. Add `pnpm smoke:factory`/`make smoke-factory`, its ephemeral-but-retained-on-failure mutability, and the full-CI versus ordinary-local/plan-only gate split without describing fake fixture details as product behavior.

## Verify

- `pnpm smoke:factory` — runs the one offline journey directly and proves its cross-process lifecycle, continuity, publication, cleanup, and replay assertions.
- `pnpm exec vitest run test/docs-contracts.test.ts` — proves command/docs inventory and the local/full/plan-only gate split without joining the smoke to Vitest.
- `pnpm check:ci` — canonical full gate; must complete the existing `pnpm check` work before running the Factory smoke.

## Boundaries

- Test/development surface only: no public Factory command, production hook, lifecycle relaxation, imported-result path, scheduler, provider registry, generic fixture framework, or additional CI workflow.
- Item-file authority only. No Linear selector, `--apply`, live provider, live GitHub, network, public internet, or user Factory-store access.
- Do not add plan PR publication/merge to this smoke; focused existing tests retain that responsibility under the accepted clarification.
- Do not add failure injection or failed-role-only review recovery; FER-77 owns those paths.
