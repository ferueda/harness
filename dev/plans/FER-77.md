# Resume only incomplete Factory implementation reviewers

## Goal

Make `reviewImplementationCandidate` retain each successful implementation or quality review in durable Factory evidence and, after a retryable reviewer failure, invoke only the fixed role that still lacks valid evidence. Reuse must survive a new CLI process and remain bound to the implementation phase run, stable review round, candidate attempt, base/candidate commit and tree, exact persisted reviewer execution profile, current per-role review contract, and content-addressed rendered prompt/output. Factory may publish `implementation.review.completed`, promote a passing candidate, or expose `needs_changes`/`blocked` only after both roles validate and are combined into the existing immutable `partial: false` manifest.

Preserve Git-authority checks, one-action-per-command recovery, action-result idempotency, lifecycle/reaction behavior, continuation, publication, CAS, and Linear projection behavior. The durable Factory store owns retry reuse; unchanged `change-review` remains the provider-neutral subset executor.

## Changes

1. `lib/factory-implementation-review-checkpoint.ts` (new), `lib/prompts/factory-implementation.ts` — own a Factory-only, strict versioned cumulative-checkpoint contract and its validation helpers. Bind the checkpoint to phase run, review round (the existing retry reaction preserves this action attempt), candidate attempt, base, candidate commit/tree, and `FactoryActionExecutionProfileSchema`; allow only optional fixed `implementation` and `quality` entries, each containing `FactoryArtifactRef`s for the rendered `*-review.prompt.md` and schema-valid `*-review.json` plus a SHA-256 role-contract fingerprint. Compute that fingerprint from a canonical descriptor containing the role id, current unrendered role prompt, exact `schemas/review-output.schema.json` bytes, an explicit terminal-verdict-contract version, and an exported Factory implementation-review handoff-contract version. This keeps it stable across action-local artifact paths while invalidating reuse when prompt, output, verdict, or handoff semantics change.

   Load a checkpoint only from the immediately causative matching retryable `reviewImplementationCandidate` failure. Verify its `failure.json` artifact, follow the checkpoint ref recorded there, require that ref in the event's evidence, then verify every checkpoint/prompt/output hash before parsing content. A phase/review/candidate/base/tree/profile mismatch makes the whole checkpoint incompatible and reusable roles empty; a per-role contract mismatch invalidates only that role. Hash failure or a current-version checkpoint/role entry that claims successful evidence but is malformed, missing, or schema-invalid is corrupted evidence and must fail terminally before provider invocation. Unsupported checkpoint versions are incompatible rather than migration input; do not mutate old evidence.

2. `lib/factory-implementation-review-evidence.ts:validateImplementationReviewEvidence` — split full single-run metadata validation from cumulative role aggregation. Keep the existing full-run path for ordinary complete change-review results, and add a path that parses the two authenticated role outputs, enforces the same terminal verdict/must-fix rules, computes the existing aggregate `pass`/`needs_changes`/`blocked` verdict, and collects blocking findings without claiming both outputs came from one workflow run.

3. `lib/factory-implementation-review-action.ts:runLeased`, `fail`, completed-result recovery — after candidate validation, authenticate any compatible prior checkpoint and select missing roles in fixed workflow order. Invoke `runChangeReview(reviewCtx, { steps: missingRoles })`; leave its subset metadata honestly `partial: true`. From the current staged run, add a role only when metadata identifies it as successfully completed and its prompt/output artifacts exist and validate. A role with transport failure, abort, malformed output, or no successful metadata gets no entry; metadata that falsely claims success without valid artifacts fails closed. Merge new entries with reused entries and durably write a new cumulative checkpoint in the current action directory so staged-result crash recovery remains action-local and never reinvokes completed work.

   If a role is still missing, preserve the existing retryable or abort failure classification and write the checkpoint ref into `failure.json` and `data.evidence`; the next manual command follows the existing retry reaction and reloads it. Once both roles exist, aggregate their refs—even when they originate in different action-local review runs—into the existing `review-evidence.json` shape with `partial: false`, then use the existing blocking-findings, Git-authority, promotion, lifecycle-event, and action-result paths. Append the checkpoint ref after the existing positional manifest/reviewer/blocking evidence refs on the completed event, and extend recovered-result validation to authenticate its bindings and role refs. Do not synthesize a successful change-review run, copy prior results into workflow context, or change lifecycle/state-machine schemas.

4. `test/factory-implementation-actions.test.ts`, `lib/factory-implementation-review-checkpoint.test.ts` (new) — update the existing fake review-run writers to emit the prompt/output artifacts and successful-role metadata now authenticated by Factory. Replace the whole-review retry expectation with action journeys in both failure directions: one role succeeds and the other fails retryably, reopen the persisted implementation context to model process restart, retry, assert the workflow receives only the missing `steps`, and prove the successful provider/output is not repeated. Cover final cumulative `pass`, `needs_changes`, and `blocked` aggregation, no partial completion/promotion, and unchanged staged/action-result recovery. At the focused checkpoint seam, cover whole-checkpoint incompatibility for phase/review/candidate commit/tree/attempt/profile changes, affected-role invalidation for contract changes, no entry for failed/aborted/malformed/missing role results, and terminal rejection without invocation for checkpoint/prompt/output tampering. Keep overlapping assertions out of the action journeys.

5. `scripts/smoke-factory.ts`, `docs/contributing/factory.md` — extend the offline multi-process Factory journey so the fake provider completes the implementation reviewer, fails quality transport once, exits the command, and succeeds on the next separately spawned manual command. Assert the retryable failure and checkpoint evidence, unchanged candidate, implementation invocation once versus quality twice for that candidate, complete aggregate, and unchanged downstream continuation/publication. Update the contributor guide's present-tense implementation-review/evidence ownership: Factory authenticates cumulative role evidence and selects missing roles; change-review only executes the requested subset.

## Boundaries

- No generic review cache/import/resumability API, new scheduler/store/event/handler/state, automatic retry loop, operator role selector, CLI retry policy, or Inngest-owned reuse decision.
- Reuse applies only to Factory implementation review and only through matching retryable-failure evidence. Never treat a partial role set as Factory completion or promotion evidence.
- Do not change `workflows/change-review.workflow.ts` behavior or fabricate full-run metadata; its existing `steps` seam remains the execution boundary.
- STOP rather than reuse when artifact integrity cannot be authenticated. Incompatible identity/profile/contract evidence may be ignored and rerun; corrupted referenced evidence must fail closed.
