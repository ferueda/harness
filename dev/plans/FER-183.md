# Bound Factory-owned automatic retry chains

## Goal

Stop one Factory action from deriving automatic retries forever. Each newly
created phase run snapshots a Factory-owned policy whose default allows three
executions for one `(phaseRunId, handler, attempt)` chain. The initial execution
counts as one; only retryable executions one and two may produce another invoke
reaction. Execution three must persist an effective human-required failure with
a clear ceiling reason, reduce to `needs-human`, and produce no next operation.

Only distinct authenticated Factory action results in the durable lifecycle
count. Inngest retries of thrown infrastructure errors, duplicate delivery,
saved-result recovery, and next-event send retries must reuse the same action
identity without consuming the ceiling. Success, a different phase run,
handler, or attempt, and an explicit durable human continuation break the
chain. CLI and hosted paths must derive the same state and reaction, while
existing Linear attention projections remain repairable and idempotent.

## Changes

1. `lib/factory-phase-run.ts:FactoryPhaseRunV2BaseSchema`,
   `lib/factory-lifecycle-events.ts:RequestData`, and
   `lib/factory-phase-request.ts:phaseRequestEvent` — define one strict
   `FactoryAutomaticActionPolicy` with `maxExecutions`, default it to `3` when
   creating a phase, and copy the immutable value into the corresponding
   `*.requested` lifecycle event. Update
   `lib/factory-run-context.ts:createFactoryRunContextInternal`,
   `lib/factory-planning-run-context.ts:createFactoryPlanningRunContext`, and
   `lib/factory-implementation-run-context.ts:createFactoryImplementationRunContext`
   so every new triage, planning, and implementation phase snapshots the
   default before its request is appended; duplicate request authentication
   must require the phase context and lifecycle copy to agree. Make the added
   fields optional only at the read boundary so existing phase-run and request
   evidence remains valid; an old phase with no snapshot keeps its existing
   replay behavior, while any restart or new phase receives the bounded policy.
   This additive cutover does not change `FACTORY_STORE_FORMAT`.

2. Add `lib/factory-action-failure.ts` as the single effective-failure
   classifier used by `workflows/factory-triage.workflow.ts`,
   `lib/factory-plan-candidate-action.ts`,
   `lib/factory-plan-review-action.ts`,
   `lib/factory-implementation-candidate-action.ts`, and
   `lib/factory-implementation-review-action.ts`. For a proposed retryable
   result, read the snapshotted policy from the active phase request and walk
   only the direct causal sequence of prior `factory.action.failed` events with
   the same phase run, handler, and attempt; count the current execution once.
   A different event type or identity ends the scan, so success, continuation,
   handler/attempt changes, and phase changes reset it without another counter.
   Below the maximum, retain the handler's proposed classification. At the
   maximum, return `human-required`, a stable explanatory message containing
   the execution count and limit, and structured `retryCeiling` data. Apply this
   result before publishing each failure evidence file and
   `action-result.json`, including triage `meta.json`, so the immutable evidence,
   authenticated result, and lifecycle event agree. Keep provider-specific raw
   failure diagnosis in its existing staged evidence; do not add counting to
   individual handlers.

3. `lib/factory-lifecycle-events.ts:factory.action.failed`,
   `lib/factory-action-result-auth.ts:authenticateFailure`, and
   `lib/factory-state-machine.ts:reduceFailure` — validate the structured
   ceiling only on an effective human-required failure and authenticate it
   against the failure artifact. Reduce a ceiling failure to `needs-human` even
   when it retains a reusable planning or implementation candidate, retain the
   candidate/review identity plus the ceiling reason in rebuildable state, and
   return the existing human wait reaction so no operation request is derived.
   Ordinary retryable, terminal, human-required, and retained-candidate
   behavior remains governed by the existing reducer paths.

4. `lib/factory-continuation.ts:observeFactoryContinuation` and
   `recordFactoryContinuation`, together with
   `lib/factory-state-machine.ts:continuationAllowed` — allow a candidate-bound
   `needs-human` state caused by `retryCeiling` to accept the same explicit
   `revise` or `re-review` continuation contract used by other human waits.
   Clear the ceiling marker when that event reduces and derive the next
   candidate/review attempt normally, which starts a new causal chain without
   weakening stale-cursor, candidate, review, or response-artifact checks.
   Triage and failures without a reusable candidate continue through an
   explicitly authorized phase restart.

5. `bin/factory-commands.ts:runFactoryTriageCommand`,
   `bin/factory-planning-cli.ts:applyTerminalPlanningProjection`, and
   `bin/factory-implementation-cli.ts:repairTerminalProjection` — select
   terminal projection behavior from the effective durable failure/state, not
   the provider's original retryable diagnosis. Project a ceiling as human
   attention with its persisted reason, and preserve the existing run-scoped
   Linear status/comment markers so a retry repairs a partial projection
   without rerunning the handler or adding a duplicate comment.

6. Add focused regressions at the existing seams:
   - `test/factory-action-kernel.test.ts` and
     `test/factory-continuation.test.ts` cover the policy/request snapshot,
     backward-compatible evidence, the third-failure `needs-human` state,
     retained-candidate continuation, and resets after success, a new
     handler/attempt/phase, or explicit continuation.
   - A colocated `lib/factory-action-failure.test.ts` exercises all five Factory
     handlers through the shared classifier and proves that three retryable
     outcomes yield exactly two automatic invoke reactions.
   - `lib/factory-operation.test.ts`,
     `lib/factory-hosted-operation-recovery.test.ts`, and
     `lib/factory-inngest-adapter.test.ts` prove saved-result recovery,
     duplicate delivery, Inngest step retry, and next-event retry reuse the
     same durable result, while the third distinct result has no `next`.
   - `test/factory-triage-apply-command.test.ts`,
     `test/factory-planning-cli.test.ts`, and
     `test/factory-implementation-actions.test.ts` prove CLI reactions match the
     hosted boundary and repeated human-attention projection remains
     idempotent for triage, producer, and reviewer failures.

7. `docs/contributing/factory.md` and
   `skills/factory-operator/SKILL.md` — document the Factory-owned automatic
   retry ceiling, its durable chain identity and reset rules, and the required
   stop at human attention. Keep it explicitly separate from Inngest's
   infrastructure retries and from caller-controlled, unlimited human
   review/revision cycles.

## Boundaries

- Do not add a public config or CLI override for the three-execution default in
  this work item.
- Do not reinterpret or backfill phase contexts that predate the policy
  snapshot, and do not bump or migrate the Factory store format.
- Do not move retry policy into Inngest, add an adapter loop or scheduler,
  change its three infrastructure retries or 110-minute action deadline, or
  implement FER-100 supervisor/fixer recovery.
- Do not cap human-authorized `revise` or `re-review` cycles or count
  `needs_changes` continuations as automatic retries.
