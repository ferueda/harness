# Bound Factory-owned automatic retry chains

## Goal

Stop one Factory action from deriving automatic retries forever. Each new phase
run snapshots a Factory-owned policy whose default allows three executions for
one `(phaseRunId, handler, attempt)` chain. The initial execution counts as one;
only retryable executions one and two may produce another invoke reaction.
Execution three must persist the existing `human-required` failure shape with a
clear ceiling message and produce no next operation.

Only distinct authenticated Factory action results in the durable lifecycle
count. Inngest retries of thrown infrastructure errors, duplicate delivery,
saved-result recovery, and next-event send retries must reuse the same action
identity without consuming the ceiling. Success, a different phase run,
handler, or attempt, and an explicit durable human continuation break the
chain. CLI and hosted paths must derive the same state and reaction, while
existing Linear attention projections remain repairable and idempotent.

## Changes

1. `lib/factory-phase-run.ts:FactoryPhaseRunV2BaseSchema` — define one strict
   `FactoryAutomaticActionPolicy` with `maxExecutions` and store it only in the
   immutable phase-run identity. Update
   `lib/factory-run-context.ts:createFactoryRunContextInternal`,
   `lib/factory-planning-run-context.ts:createFactoryPlanningRunContext`, and
   `lib/factory-implementation-run-context.ts:createFactoryImplementationRunContext`
   so every new triage, planning, and implementation phase snapshots
   `{ maxExecutions: 3 }`. Keep the field optional at the read boundary: an old
   phase with no policy keeps its existing retry behavior, while any restart or
   new phase receives the ceiling. Do not copy the policy into `*.requested`
   events or add another agreement check; action execution already
   authenticates and opens the phase-run identity. This additive cutover does
   not change `FACTORY_STORE_FORMAT`.

2. Add `lib/factory-action-failure.ts` as the single effective-failure
   classifier used by `workflows/factory-triage.workflow.ts`,
   `lib/factory-plan-candidate-action.ts`,
   `lib/factory-plan-review-action.ts`,
   `lib/factory-implementation-candidate-action.ts`, and
   `lib/factory-implementation-review-action.ts`. For a proposed retryable
   result, read the snapshotted policy from the active phase identity and walk
   only the direct causal sequence of prior `factory.action.failed` events with
   the same phase run, handler, and attempt. Count the current execution once.
   A different event type or identity ends the scan, so success, continuation,
   handler/attempt changes, and phase changes reset it without another counter.
   Below the maximum, retain the handler's proposed classification. At the
   maximum, return `human-required` with a stable message that includes the
   execution count, limit, and original failure summary. Apply this effective
   result before publishing each failure evidence file and
   `action-result.json`, including the action-bound triage failure metadata, so
   evidence, result, and lifecycle event agree. Keep the provider's raw
   diagnosis in its existing staged evidence; do not add counters to handlers
   or lifecycle state.

3. Reuse the current failure and continuation contracts without adding a
   `retryCeiling` event field, reducer state, or special continuation path. A
   ceiling failure without a candidate already reduces to `needs-human`; one
   retaining the active planning or implementation candidate already reduces
   to `awaiting-continuation`. Both yield the existing human wait reaction.
   Existing `revise` or `re-review` continuation starts a new causal chain, and
   triage or candidate-less failures continue through an authorized phase
   restart. Keep the current artifact authentication of `failureKind` and
   message unchanged.

4. `bin/factory-commands.ts:runFactoryTriageCommand`,
   `bin/factory-planning-cli.ts:applyTerminalPlanningProjection`, and
   `bin/factory-implementation-cli.ts:repairTerminalProjection` — use the
   effective durable failure event rather than the provider's raw diagnosis.
   Extend planning projection and repair to cover a human-required failure that
   retained a candidate and therefore reduced to `awaiting-continuation`;
   implementation already has this distinction. Project the persisted ceiling
   message as human attention and preserve the existing run-scoped markers so a
   retry repairs a partial projection without rerunning the handler or adding a
   duplicate comment.

5. Add focused regressions at the existing seams:
   - `lib/factory-action-failure.test.ts` covers legacy phases, executions one
     through three, the stable human-required result, and resets after a
     different event, handler, attempt, phase, or explicit continuation.
   - `test/factory-triage-apply-command.test.ts`,
     `test/factory-planning-actions.test.ts`, and
     `test/factory-implementation-actions.test.ts` prove that the five action
     boundaries use the shared classifier before publishing immutable failure
     evidence. They also prove existing triage and implementation attention
     projection remains idempotent and retained-candidate continuation keeps
     using the current state path.
   - `lib/factory-hosted-operation.test.ts` and
     `lib/factory-inngest-adapter.test.ts` prove that the third distinct durable
     result has no `next`, while saved-result recovery, duplicate delivery, and
     an Inngest step retry do not create another counted failure.
   - `test/factory-planning-cli.test.ts` proves the newly covered
     retained-candidate planning failure projects and repairs human attention
     without another handler execution.

6. `docs/contributing/factory.md` and
   `skills/factory-operator/SKILL.md` — document the Factory-owned automatic
   retry ceiling, its durable chain identity and reset rules, and the required
   stop at human attention. Keep it explicitly separate from Inngest's
   infrastructure retries and from caller-controlled, unlimited human
   review/revision cycles.

## Boundaries

- Do not add a public config or CLI override for the three-execution default in
  this work item.
- Do not copy the policy into lifecycle request events, add a `retryCeiling`
  lifecycle/state field, or change the existing continuation contract.
- Do not reinterpret or backfill phase identities that predate the policy
  snapshot, and do not bump or migrate the Factory store format.
- Do not move retry policy into Inngest, add an adapter loop or scheduler,
  change its three infrastructure retries or 110-minute action deadline, or
  implement FER-100 supervisor/fixer recovery.
- Do not cap human-authorized `revise` or `re-review` cycles or count
  `needs_changes` continuations as automatic retries.
