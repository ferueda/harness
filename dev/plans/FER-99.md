# Record hosted Factory authority before operation delivery

## Goal

Add two trusted, host-neutral application boundaries: one starts or restarts `triage`, `planning`, or `implementation`, and one records an explicit planning or implementation `revise`/`re-review` continuation. Both must validate the caller's observed Factory identity, persist the existing `*.requested` or `factory.continuation.recorded` authority event, then use `reconcileFactoryOperations` to derive and send only the canonical identifier-only operation hint and return its single-target reconciliation result. A delivery failure therefore leaves the event log authoritative and lets the same operation be rediscovered without creating duplicate authority.

Phase starts must retain current CLI eligibility and context semantics while selecting one immutable baseline: snapshot the configured controller repository's `HEAD^{commit}` before Grove acquisition, derive the lease from that SHA and the verified predecessor event, create and validate the phase context in the acquired checkout, and compare-append the request against the same predecessor. `baseRef` remains trusted pull-request target metadata; neither application input accepts a caller-selected `baseSha`, handler, action, or idempotency key. No lifecycle lock may span Grove or delivery.

## Changes

1. Add `lib/factory-phase-request.ts` as the shared lifecycle boundary for phase-request preparation and append. Define a closed discriminated phase input and explicit observed predecessor, and expose focused `prepareFactoryPhaseRequest` / `appendPreparedFactoryPhaseRequest` operations used by both hosted and CLI adapters.

   - Preparation reads canonical events, authenticates the project/work-item identity, and rejects a stale observed cursor before workspace work. Preserve the existing deterministic `work_item.imported:<key>` event for a new item; append or reuse it before Grove only when needed to establish the non-null phase generation.
   - Centralize the effective phase rules already enforced by the CLI and reducer: triage restarts only after routed, parked, needs-human, or failed; planning restarts from needs-human or failed and may resume its exact pending restart request; implementation restarts from needs-human or failed only without a reusable candidate and may resume its exact pending restart request. A retryable action reaction remains part of its current phase run and can never satisfy restart preparation.
   - Treat an already-durable request with the same phase, intent, observed predecessor, work-item evidence, and phase options as an exact duplicate and return that phase run for reconciliation. Reject a different request or a cursor that has advanced beyond it. For a new request, re-read and validate the prepared predecessor after context creation, load the phase-run identity from the trusted store, derive all request fields and input refs from that immutable context, and compare-append `triage.requested`, `planning.requested`, or `implementation.requested`. Keep `appendFactoryActionEvent` and reducer transition validation as the final conditional-write checks; do not introduce another lock or state source.

2. Extend `lib/factory-continuation.ts:recordFactoryContinuation` so it accepts `workItemKey` instead of a mutable work-item payload and requires observed `{ expectedPredecessor, phaseRunId, candidateEventId, reviewEventId? }`. Validate those values against the current reduced state, the matching candidate/review event types, the stored phase-run identity, and existing artifact refs before writing the bounded response artifact or attempting the append. Preserve the current response hash and `factory.continuation.recorded:<digest>` identity. If that exact event and verified response artifact already exist, return its canonical state/reaction as an idempotent duplicate; reject changed decisions, response bytes, missing reviews, stale predecessors, or phase/candidate/review mismatches. Update synchronous CLI callers and existing continuation test helpers to derive the canonical key, read the current identities immediately, and pass them explicitly, leaving CLI flags and output unchanged.

3. Add `lib/factory-hosted-authority.ts` with strict Zod request schemas and inferred readonly types for `requestHostedFactoryPhase` and `recordHostedFactoryContinuation`. Close the phase request to `{ projectId, workItem, phase, intent, expectedPredecessor }` and the continuation request to `{ projectId, workItemKey, phase, decision, response, expectedPredecessor, phaseRunId, candidateEventId, reviewEventId? }`. Keep store paths, Grove configuration, existing Factory agent execution profiles/provider controls, `baseRef`, and the delivery callback in trusted runtime input rather than in the serializable caller request.

   - Authenticate request project/repository/work-item identity against `FactoryStoreMeta` and the configured controller, then run the shared preparation. Exact already-recorded duplicates skip phase creation and proceed directly to reconciliation.
   - For a new phase request, snapshot the controller's full `HEAD^{commit}` once, derive `FactoryGroveWorkspaceIntent` with the prepared predecessor as `phaseGeneration`, and acquire/reacquire Grove without retaining a lifecycle lock. Validate the returned lease and intent plus the live checkout's repository, exact starting `HEAD`, and detached/deterministic branch target before creating a run. Use the existing `createFactoryRunContext`, `createFactoryPlanningRunContext`, or `createFactoryImplementationRunContext` with trusted role/config values and `resolveFactoryImplementationInput` where applicable. Require the persisted version-2 Git identity to match the snapshotted SHA and Grove target before the shared conditional append. Export the existing Grove-intent/phase-Git comparator from `lib/factory-grove-workspace.ts` and reuse it here and in `lib/factory-hosted-operation.ts` so phase creation and later execution authenticate one baseline without parallel matching rules.
   - After either phase append or `recordFactoryContinuation`, call `reconcileFactoryOperations` for exactly `{ projectId, workItemKey, factoryStore }` and the injected delivery callback, and return its one result. Do not construct or accept an operation hint directly. Delivery is therefore impossible before durable authority; a callback failure returns the reconciler's bounded `attention` result while the event remains available to FER-98, and a state change before the reconciler's final read returns its canonical wait/stale outcome.

4. Route the phase-start sections of `bin/factory-commands.ts:runFactoryTriageWithLinearApply`, `bin/factory-planning-cli.ts:runOneFactoryPlanningAction`, and `bin/factory-implementation-cli.ts:runOneFactoryImplementationAction` through the shared preparation/append rules. The CLI remains the synchronous non-Grove adapter: it reads the observed identity immediately, creates the phase context in its already-provisioned workspace, conditionally appends, then retains its current direct action execution, Linear projection, output, and recovery behavior. Remove only the duplicated restart/pending-request checks and event construction now owned by `lib/factory-phase-request.ts`; do not add flags or make a provider/action retry create a new phase run.

5. Add `lib/factory-hosted-authority.test.ts`, reusing or extending `lib/factory-hosted-operation-test-fixtures.ts` for isolated real Git repositories, Factory stores, and typed Grove/delivery fakes. Prove at the new application seam:

   - phase start/restart and continuation each append their canonical event before the delivery callback receives the exact identifier-only request;
   - all three phase eligibility rules, including pending restart convergence and implementation's reusable-candidate rejection;
   - stale/out-of-order predecessor and continuation identities fail before Grove, response persistence, or delivery, while compare-and-append rejects a state change injected during acquisition;
   - exact duplicate phase and continuation calls reuse one authority event and operation, while divergent duplicates fail closed;
   - the controller snapshot, Grove intent/checkout, version-2 phase context, and request predecessor agree, and a changed HEAD, repository, lease intent, or target prevents request append and delivery;
   - a delivery callback that throws returns `attention` after the mutation, and a later direct `reconcileFactoryOperations` call resends the same operation without adding another authority event.

   Update focused continuation and CLI tests (`test/factory-continuation.test.ts`, `test/factory-triage-apply-command.test.ts`, `test/factory-planning-cli.test.ts`, and `test/factory-implementation-actions.test.ts`) for the explicit observed identities and to confirm existing CLI restart/projection behavior still uses new phase-run IDs. Retain `lib/factory-hosted-operation.test.ts` coverage that later delivery reauthenticates the same phase request and Grove identity.

Keep the real controller repository, Grove pool/setup hook, Inngest delivery, hosted runner in a fresh process, terminal release, and post-release recovery. Assert the boundary-selected controller SHA is the lease/context base, the request is durable before the Inngest event is observed, and the existing replay still performs no second provider run or setup.

7. Update `docs/contributing/architecture.md`, `docs/contributing/factory.md`, and the smallest matching assertions in `test/docs-contracts.test.ts` to describe the new authority-before-delivery flow and ownership. Make `lib/factory-hosted-authority.ts` responsible for trusted observed requests, pre-request Grove/context creation, durable authority, and reconciliation; keep `lib/factory-hosted-operation.ts` responsible only for executing an already-authorized identifier-only operation. State that the CLI is a synchronous adapter, async adapters must preserve observed identities, transport authentication remains adapter-owned, and hosted publication/merge authority is still outside this boundary.

## Boundaries

- Do not add credentials, users, authorization roles or actors, authorization callbacks, HTTP/webhook adapters, project discovery, scheduling, a command/handler registry, multi-worker fencing, or a caller-generated idempotency key.
- Do not add or change lifecycle event types, operation request/receipt fields, an outbox, delivery state, or a second workflow source of truth. Existing event IDs, compare-and-append, action identity, and deterministic Inngest delivery IDs remain the idempotency layers.
- Do not include plan/implementation publication, pull-request side effects, merge acknowledgement, tracker-derived authority, Grove terminal release/repair, or automatic continuation choice. Hosted publication remains FER-184.
- Do not accept `baseSha` or an operation/handler from application callers. A different baseline must be provisioned or pinned at the controller before the call, and delivery may only use the operation recomputed by reconciliation after durable authority.
