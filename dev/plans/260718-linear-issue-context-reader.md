# Build a standalone Linear issue-context reader

## Goal

Implement [FER-213](https://linear.app/ferueda/issue/FER-213/build-a-standalone-linear-issue-context-reader) as one read-only `lib/linear` capability. A caller outside Factory can load one Linear issue by human identifier or opaque ID and receive bounded, stable, JSON-safe context for later triage. The result includes comments, a shallow parent, shallow children, current-centric duplicate/blocker/related references, attachment link metadata, and honest completeness flags. After reviewing FER-213, the user explicitly approved adding bounded shallow children and child completeness on 2026-07-18; that decision supersedes the issue's original child exclusion. The existing Factory adapter remains unchanged.

## Changes

1. `lib/linear/client.ts:createLinear`, `createLinearForClient`, and exported context types — wrap the installed `@linear/sdk` behind a narrow read-only client seam. Require positive collection limits for comments, labels, relations, attachments, and children. Resolve both `TEAM-123` and opaque IDs through bounded exact `issues` filters so zero and ambiguous matches have the same behavior. Expose one `LinearReadError` with exact codes: `invalid-config` for bad limits, `invalid-reference` for an empty reference, `not-found` for zero exact matches, `ambiguous-reference` for multiple exact matches, `invalid-response` for missing required SDK data or impossible relation cardinality, and `upstream` for SDK request failures; every wrapped failure preserves `cause`.
2. `lib/linear/client.ts:getIssueContext` — return issue ID, identifier, title, description, URL, required `{ id, name, type }` state, required `{ id, key, name }` team, nullable `{ id, name, url }` project, nullable assignee and creator, labels, comments, parent, children, duplicate target, blockers, related issues, attachments, and ISO issue timestamps. Optional values are explicit `null`; collections always exist. A user is `{ id, name, displayName }`. A comment actor is the discriminated union `{ kind: "user", id, name, displayName }`, `{ kind: "bot", id: string | null, name: string | null }`, `{ kind: "external", id, name: null }`, or `null`; nullable bot identity fields preserve valid SDK bot responses rather than rejecting or discarding them. A comment is `{ id, body, author, parentId, quotedText, createdAt, updatedAt }`; a label is `{ id, name }`.
3. `lib/linear/client.ts:getIssueContext` — read comments, labels, outgoing/inverse relations, attachments, and children with explicit Relay cursors until each configured limit. A collection is truncated whenever reading stops at its limit, including when the returned count exactly equals the limit; otherwise use Linear's `hasNextPage`. Deduplicate issue references by ID before sorting. Sort comments by `(createdAt, id)`, labels by `(name, id)`, children/blockers/related references by `(identifier, id)`, and attachments by `(createdAt, id)`, using ascending code-point comparisons on normalized strings. Resolve unique comment users, referenced issues, and referenced workflow states in bounded bulk queries so collection size does not create one SDK request per node.
4. `lib/linear/client.ts:getIssueContext` — keep graph data shallow. Parent, children, and relation targets use `{ id, identifier, title, url, state }`. Map one outgoing `duplicate` relation to nullable `duplicateOf`; map incoming `blocks` relations to `blockedBy`; merge outgoing and incoming `related` relations into a deduplicated `related` array. Outgoing `blocks` and incoming `duplicate` describe work affected by the current issue and are not projected into these current-issue triage fields. More than one outgoing duplicate is `invalid-response`. Attachments expose only `{ id, title, subtitle, url, sourceType, createdAt, updatedAt }`; never expose or download `bodyData`, `metadata`, or `source`.
5. `lib/linear/client.ts:getIssueContext` — return `commentsTruncated`, `labelsTruncated`, `relationsTruncated`, `attachmentsTruncated`, and `childrenTruncated`. `relationsTruncated` is true if either outgoing or inverse relation pagination truncates. The result contains no SDK model, promise, `Date`, or `undefined` value.
6. `lib/linear/client.test.ts` — prove the complete public service result through a fake client: human and opaque lookup; base fields; present and absent project, assignee, and creator; lazy required relations; multipage reads; conservative exact-limit truncation; the exact collection comparators above and JSON serialization; batched author/reference/state hydration; user, external, complete-bot, and nullable-bot comment actors; parent and bounded children; relation direction and duplicate cardinality; the attachment allowlist; invalid limits; lookup failures; upstream causes; and missing required state/team. Add lower pagination assertions only where the public result cannot distinguish cursor or over-fetch errors.
7. `docs/contributing/architecture.md:Source map` — record `lib/linear/` as the standalone Linear communication boundary, distinct from the Factory-owned projection files.

## Boundaries

- Do not import Factory, Inngest, provider, prompt, or triage code from `lib/linear`.
- Do not migrate or make Factory consume the new module in this issue.
- Do not add mutations, search, recursive issue expansion, caching, retries, a generic tracker abstraction, a custom GraphQL query, or an exported JSON schema.
