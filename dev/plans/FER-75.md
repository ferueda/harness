# Inspect Durable Factory Work Items Without Advancing Them

## Goal

Add `harness factory inspect` for one `--linear-issue` or `--item-file`. It must derive the canonical durable work-item key, rebuild current state from the canonical lifecycle log, compute the existing Factory reaction, and print one deterministic JSON object in this order: `workItemKey`, `artifactRoots`, `state`, `latestEvent`, `reaction`. `artifactRoots` maps `repository` to the resolved workspace and `factory-store` to the resolved Factory project root, matching artifact-ref bases; the latest event and every nested `{ base, path, sha256 }` ref remain verbatim. With no history, the last three fields are `null` rather than a fabricated idle state.

The reaction gains an exact shell command only when the persisted state can mechanically select the next station. One canonical decorator must drive inspection and triage/planning/implementation station output so identical durable state and selector provenance produce the same command. Commands retain the workspace and explicit Factory store overrides; Linear station commands include `--apply`. Human, plan-merge, complete, failed, and stale-event waits have no invented command. Inspection performs no lifecycle/store writes, tracker reads or mutations, provider/reviewer work, or continuation.

Acceptance: active triage, planning, implementation/revision, human wait, failure, and terminal completion report the same state/latest event/reaction the existing kernel produces; handler and attempt remain visible on invoke reactions; relevant evidence stays addressable through the two roots and verbatim refs; unchanged repeated invocations produce byte-identical stdout and leave durable artifacts unchanged.

## Changes

1. `lib/factory-lifecycle-kernel.ts:readFactoryActionEvents` — add an explicit inspection read mode that reuses store-format validation, canonical JSONL parsing, action-identity validation, and reduction inputs without acquiring/removing the current filesystem lock. Preserve the existing locked default for station callers; inspection must not create a store marker, lock directory, state projection, or any other path. Cover this distinct zero-write invariant in `test/factory-action-kernel.test.ts`, including absent history and an existing event log.

2. `lib/factory-inspect.ts:inspectFactoryWorkItem` — add the host-usable inspector keyed only by canonical `workItemKey`. Read events through the non-mutating mode, reduce with `reduceFactoryLifecycleEvents`, select `events.at(-1)`, and call `decideNextFactoryAction` only when both state and latest event exist. Return the stable root object and normalize missing history to `state: null`, `latestEvent: null`, and `reaction: null`; do not read the rebuildable state JSON, verify/crawl artifact contents, flatten evidence, or derive truth from an item file or tracker metadata.

3. `bin/factory-manual-command.ts`, `bin/factory-action-output.ts`, `bin/factory-commands.ts:addFactoryCommands`, `bin/factory-planning-cli.ts`, and `bin/factory-implementation-cli.ts` — add one canonical selector-aware command builder/decorator, then use it for both inspection and every station result path, including terminal recovery and already-waiting returns. It receives the pure reaction, reduced state when present, and command provenance. Invoke reactions use their existing phase; `phase-command` waits map imported idle state to triage, routed triage state to planning or implementation, and approved planning state to implementation. It emits `harness factory triage` or the appropriate `planning run` / `implementation run`, one original selector, workspace, and explicit store overrides in one stable order with one shell-quoting implementation. Linear selectors always add `--apply`, independent of whether the current invocation constructed an apply adapter. It strips or replaces any persisted triage command and leaves human, plan-merge, complete, failed, stale-event, and null reactions commandless.

   Register `factory inspect` with `--workspace`, both existing selector flags, and the existing Factory store override flags, but no apply/rerun/provider/mutation options. Reuse `validateFactoryWorkItemInput`; for `--item-file`, reuse `assertFactoryItemFileExists`, `readFactoryWorkItemFile`, and `deriveFactoryWorkItemKey` solely to derive the key while retaining the supplied path as command provenance. For `--linear-issue`, reuse `parseLinearIssueIdentifier`, normalize only the durable key to `linear:TEAM-123`, retain the selector for command parity, and reject UUIDs or other opaque values with a store-only/no-fetch error; do not resolve Linear settings, require `LINEAR_API_KEY`, or construct an adapter. Decorate only the inspector's returned `reaction`, never `state` or `latestEvent`.

   Cut over triage, planning, and implementation coordinators to the shared decorator while each has the authoritative reduced state, threading selector/store provenance through normal, retry, recovery, and waiting exits. Then remove `factoryTriageManualCommand`, `planningCommand`, `implementationCommand`, their three private `shellArg` variants, and the invoke-only `withManualCommand`; `formatFactoryActionOutput` remains the owner of the existing stdout envelope.

## Boundaries

- No public `--work-item-key`, second inspect/next subsystem, alternate state model, event or artifact crawl, artifact-content inlining, or hash removal.
- No tracker status/comment transition input, Linear fetch, automatic continuation, rerun suggestion, polling, scheduler, registry, DAG, or hosted-orchestrator layer.
- Preserve locked event reads for existing station paths; only the explicit inspection mode may bypass the mutating lock lifecycle.
