# Execute one authenticated Factory operation

## Goal

Expose one typed application operation that executes exactly one canonical `FactoryOperationRef` against a caller-supplied existing workspace and durable Factory store. The operation must reopen the phase's immutable context, confirm that the referenced invoke reaction is still the exact durable next action immediately before dispatch, call one of the five existing handlers, and return that handler's persisted lifecycle event and reduced state. Manual triage, planning, and implementation commands must use this operation without changing provider profiles, recovery, Git/session authority, projections, publication, command output, or one-action-per-invocation behavior.

## Changes

1. `lib/factory-operation.ts:executeFactoryOperation` — extend the canonical operation-identity module with a typed executor input built from `FactoryOperationRef`, `FactoryStoreMeta`, workspace, work item, runtime controls, and the existing provider/reviewer test seams. Reopen `openFactoryRunContext`, `openFactoryPlanningRunContext`, or `openFactoryImplementationRunContext` from the persisted handler phase so snapshotted profiles, immutable inputs, Git authority, session continuation, and action evidence remain owned by the current contexts and handlers; do not consult current config or tracker state. After reopening, reread lifecycle truth and require phase run, handler, attempt, and causation to match the referenced current invoke reaction even when an action-result file already exists. A current crash-recovery result must still pass through its existing handler so the terminal event is compare-and-appended without rerunning the provider, while stale, waiting, or mismatched operations fail closed before any provider or reviewer call. Dispatch through one exhaustive `switch` over all five `FactoryHandler` literals, forward only each handler's existing timeout/signal/provider/reviewer inputs, return its `{ event, state }`, and exit without selecting or executing the reaction produced by that result.
2. `bin/factory-commands.ts:runFactoryTriageWithLinearApply`, `bin/factory-planning-cli.ts:runOneFactoryPlanningAction`, and `bin/factory-implementation-cli.ts:runOneFactoryImplementationAction` — cut over only the live active-invoke branch: create the canonical operation reference from the selected reaction and phase run, then delegate context reopening, durable revalidation, recovery, and handler selection to `executeFactoryOperation`. Remove the three coordinators' direct calls/imports for `triageWorkItem`, `producePlanCandidate`, `reviewPlanCandidate`, `produceImplementationCandidate`, and `reviewImplementationCandidate`. Keep phase creation/restart policy, dry-run behavior, the phase-specific `action-started` progress record, pre-provider Linear guards and projection repair, terminal projections, triage metadata recovery, next-reaction command decoration, publication, and merge acknowledgement in their current CLI owners. Preserve the current ordering by retaining the existing phase-context/Git validation and progress/projection sequence, then let the shared operation reopen and revalidate again immediately before dispatch; derive terminal projections and CLI output from the one returned lifecycle result without following its next reaction.
3. `lib/factory-operation.test.ts` and the existing station regression seams in `test/factory-triage-apply-command.test.ts`, `test/factory-planning-cli.test.ts`, and `test/factory-implementation-actions.test.ts` — add focused execution coverage using real durable contexts plus injected provider and plan/implementation reviewer functions. Prove that each handler literal reaches its existing implementation, returns the persisted terminal event/state, and leaves a newly selected follow-up handler uncalled; that exact staged-result recovery appends without reinvoking its producer; and that stale, waiting, action-key/phase/work-item mismatches invoke neither provider nor reviewer and create no new action result or lifecycle event. Keep the coordinator regressions proving pre-action projection order, recovery/projection repair, publication waits, session reuse, and separate candidate/review CLI invocations instead of duplicating those behaviors in a new framework fixture.

## Boundaries

- Do not add Grove or hosted project resolution; the operation receives an existing workspace and resolved trusted store metadata.
- Do not add a handler registry, plugin surface, generic workflow runner, lifecycle state/schema, CLI option, or new provider/session policy.
- Do not move phase creation, Linear/GitHub policy or projections, command rendering, pull-request publication, or merge acknowledgement out of their current owners.
