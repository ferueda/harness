# Add safe standalone Linear mutations

## Goal

Implement [FER-214](https://linear.app/ferueda/issue/FER-214/add-safe-standalone-linear-mutation-primitives) as one standalone `lib/linear` outcome: a serialized caller can safely project an already-made decision through idempotent comments, guarded state changes, duplicate relations, and blocker relations. Keep one public `LinearService`, preserve the merged issue-context reader's behavior, and keep Factory, Inngest, providers, prompts, and triage policy outside this module.

## Changes

1. `lib/linear/client.ts:createLinear`, `createLinearForClient`, and `LinearService` — reduce the current reader-heavy file to SDK construction and assembly of one read/write service. Preserve `getIssueContext(issueRef)` and add the exact object-input methods `findCommentMarker({ issueId, marker })`, `findWorkflowState({ teamId, name })`, `createComment({ issueId, body })`, `ensureComment({ issueId, marker, body })`, `updateIssueState({ issueId, expectedStateId, stateId })`, `ensureDuplicateRelation({ issueId, duplicateOfIssueId })`, and `ensureBlockedByRelation({ issueId, blockerIssueId })`. Move the existing context types, lookup, normalization, and context assembly into `lib/linear/read.ts` without changing the public result, while `client.ts` re-exports the existing `LinearReadLimits`, user, actor, workflow-state, issue-reference, issue-context, and read-client types from their new owner. Expand the injected factory seam intentionally: `createLinearForClient` accepts a new `LinearClientLike` with the read and mutation SDK subset needed by the full service; keep `LinearReadClient` available as the read-only subset but no longer sufficient for that full-service factory. Move the existing reader coverage to `lib/linear/read.test.ts` and continue testing through the public service factory. Use explicit type-only imports and erasable TypeScript syntax.
2. `lib/linear/error.ts:LinearError` and `lib/linear/pagination.ts:readLimited` — intentionally replace and stop exporting the currently unconsumed `LinearReadError` with one module-wide `LinearError` carrying `invalid-config`, `invalid-reference`, `invalid-input`, `not-found`, `ambiguous-reference`, `incomplete`, `conflict`, `rejected`, `invalid-response`, or `upstream`; re-export the replacement from `client.ts` and preserve upstream causes. Share the existing bounded Relay cursor behavior between context reads and pre-mutation safety scans without exporting pagination on `LinearService`; reaching an exact configured limit remains conservatively incomplete.
3. `lib/linear/read.ts:findCommentMarker` and `findWorkflowState` — add generic read operations using opaque issue/team IDs. Marker lookup returns `Promise<string | null>`: validate a non-empty marker, follow comment cursors within the configured comment limit, return the matching comment ID from any scanned page, return `null` only after a complete absent scan, and throw `incomplete` when an absent scan truncates. Workflow-state lookup queries by explicit team ID and exact non-empty name with at most two results, returning normalized `{ id, name, type }` data; zero matches are `not-found` and multiple matches are `ambiguous-reference`. Neither operation owns route policy.
4. `lib/linear/write.ts:createComment`, `ensureComment`, and `updateIssueState` — validate non-empty opaque IDs and bodies, normalize SDK payloads to plain results, translate `success: false` to `rejected`, reject malformed success payloads, and wrap SDK failures with causes. Return exactly `{ id }` from direct creation, `{ created, id }` from ensure, and `{ changed, stateId }` from state update. `ensureComment` requires its body to contain the marker and creates only after a complete absent-marker scan. Require `expectedStateId` for state changes: an issue already at the target returns `{ changed: false, stateId }`, a current state different from the expected state is `conflict` without mutation, and only an expected match is updated and returns `{ changed: true, stateId }`. This guard is best-effort because Linear has no conditional state mutation; callers still serialize per issue.
5. `lib/linear/write.ts:ensureDuplicateRelation` and `ensureBlockedByRelation` — scan the correct bounded relation direction before creating. A duplicate is `issueId -> duplicateOfIssueId`; an existing different outgoing duplicate is `conflict`. A blocker is `blockerIssueId -blocks-> issueId`; multiple unrelated blockers remain valid and untouched. Matching relations on any scanned page return `{ created: false, id }`; new relations return `{ created: true, id }`; incomplete absent scans and self-relations fail before mutation. Apply the shared mutation contract to the distinct relation payload: `success: false` is `rejected`, a successful payload without a valid relation ID is `invalid-response`, and SDK throws become caused `upstream` errors.
6. `lib/linear/write.test.ts` — prove the exact public results for matched and completely absent marker lookup; direct comment creation; created/repeated comment projection; changed/no-op state projection; and created/existing relations. Cover comment, state, and relation `success: false`, malformed-success IDs, and caused SDK failures through their distinct payload paths; first-page, later-page, repeated, and incomplete marker projection; exact workflow-state lookup; guarded state conflict; duplicate conflict and direction; multiple blocker preservation and direction; later-page relation matches; incomplete relation scans; self-relation rejection; JSON serialization; and exact typed errors. Use focused mutation-call assertions through the public fake to prove no write occurs for matched/repeated comments, already-target state, existing duplicate or blocker relations, duplicate conflicts, incomplete scans, state conflicts, self-relations, or other rejected input; keep lower cursor assertions only where the public result cannot prove later-page behavior.

## Boundaries

- Do not migrate Factory or import Factory, Inngest, provider, prompt, or triage code.
- Do not add decision-to-status mapping, retries, transactions, generic tracker or relation abstractions, labels, assignments, issue creation, projects, cycles, or initiatives.
- Do not expose an unguarded state mutation or treat marker/relation idempotency as atomic.
