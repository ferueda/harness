# Close Factory implementation through a human-merged pull request

## Goal

Extend the implementation lifecycle past automated review to a durable pull-request publication and explicit post-merge acknowledgement. A passing review must leave Factory at `awaiting-pr-publication` and Linear at configured Implementing; publishing the ready PR must move Factory to `awaiting-pr-merge` and Linear to configured Ready for Review; only a human-authorized acknowledgement of that recorded PR and a Git commit containing the reviewed candidate may make Factory `complete` and project Linear to configured Done.

Git and Factory lifecycle events remain authoritative. Harness must not push, open, poll, or merge a PR, call GitHub, infer merge state from GitHub/Linear, or run providers/reviewers from either publication command. Once the passing review exposes the publication wait, an executor with explicit publication authority may push/open/prepare the PR, including CI or review repairs, but a request to open or deliver a PR never authorizes merge.

## Changes

1. **`lib/factory-lifecycle-events.ts` and `lib/factory-state-machine.ts` — add the two implementation PR facts and make review pass non-terminal.**
   - Add strict, phase-run-scoped `implementation_pr.opened` and `implementation_pr.merged` events. The opened event records the PR URL and exact passing candidate commit; the merged event records that same URL and the acknowledged merge commit.
   - Replace implementation review-pass `complete` with `awaiting-pr-publication`; accept `implementation_pr.opened` only from that state and reduce it to `awaiting-pr-merge`; accept `implementation_pr.merged` only for the recorded URL and reduce it to `complete`. Preserve the PR URL/reviewed commit, and the merge commit after closure, in the implementation state read model so durable inspection exposes the handoff facts.
   - Add commandless `pr-publication` and `pr-merge` wait reactions. Keep `complete` exclusively behind the merge event; neither wait schedules a handler or invents a command that performs a merge.
   - Cover event validation, legal/illegal transitions, URL binding, state fields, and reaction selection in `test/factory-action-kernel.test.ts`; update `test/factory-implementation-actions.test.ts` and staged-review recovery expectations so pass still promotes the exact candidate and leaves a clean branch, but now waits for PR publication.

2. **`bin/factory-implementation-cli.ts` and new `bin/factory-implementation-publication-cli.ts` — add `publish` and `mark-pr-merged`, and transfer terminal ownership away from review pass.**
   - Register `harness factory implementation publish --linear-issue <issue> --url <pr-url> [--apply]` and `harness factory implementation mark-pr-merged --linear-issue <issue> --url <pr-url> --commit <merge-sha> [--apply]`, with the existing workspace/store overrides. Keep this implementation-specific rather than introducing a generic publication framework.
   - In the publication helper, resolve the active phase and its final passing review, follow that review's causation to the exact candidate, verify the immutable candidate/ref and persisted phase identity, and require the current persisted branch tip to contain that candidate (`git merge-base --is-ancestor`) before appending `implementation_pr.opened`. This binds only the reviewed commit while allowing later branch commits made during authorized CI/review repair; it does not inspect GitHub.
   - For merge acknowledgement, require the supplied URL to equal the durable opened URL, resolve `--commit` as a local Git commit, and require the exact recorded reviewed candidate to be its ancestor before appending `implementation_pr.merged`. Reject absent/non-commit SHAs, a different PR URL, missing publication, or a commit that does not contain the candidate; do not require or perform any provider/GitHub/merge operation.
   - Make both event IDs deterministic per phase run. Identical retries reuse the existing event and only repair projection; a changed URL, candidate binding, or merge SHA conflicts with durable truth. Append the lifecycle fact before its optional Linear projection so projection failure cannot lose or duplicate the fact.
   - Remove the review-pass call and terminal-repair path for `applyImplementationCompleted`; publication commands become the sole owners of ready-PR and merged-completion projections. Update `assertLiveImplementationStatus` for the new waits (`Implementing`, then Ready for Review, then Done) while retaining existing human/failure repair behavior.
   - Add `test/factory-implementation-publication.test.ts` using real temporary Git histories for: exact candidate binding, repair commits descending from the candidate, one opened/merged event across identical retries, conflicting retries, wrong URL, unknown/non-commit/wrong-ancestry merge SHAs, no publication, no provider work, and stable command output. Extend `test/factory-implementation-cli.test.ts`, `test/cli.test.ts`, and `scripts/smoke-dist.ts` for both public subcommands and their help.

3. **`lib/schemas.ts`, `lib/config.ts`, root `harness.json`, `test/config.test.ts`, and Linear fixtures — make the two board projections explicit configuration.**
   - Add required `factory.linear.statuses.readyForReview`; make `done` required because implementation closure now owns that station transition. Keep only `canceled` and `duplicate` optional. Do not add legacy aliases or fallback status names.
   - Add `"readyForReview": "Ready for Review"` to this repository's root `harness.json` status map so the FER-78 dogfood and all root Factory commands satisfy the new schema; retain its existing `"done": "Done"` mapping.
   - Include `readyForReview` and `done` in implementation-status distinctness validation and the targeted missing-mapping diagnostic so ambiguous or stale target-repo configuration fails before mutation.
   - In `test/config.test.ts`, load the repository root config through `resolveFactoryLinearSettings` and assert its Ready for Review/Done mappings, alongside missing-mapping and distinctness cases. Update `test/factory-linear-test-helpers.ts` plus inline `harness.json`/status fixtures in affected CLI tests for the new required keys.

4. **`lib/factory-linear-implementation-apply.ts`, new `lib/factory-linear-implementation-handoff.ts`, and `lib/factory-linear-adapter.ts` — project publication and closure idempotently.**
   - Retain start, attention, and failure behavior in the existing implementation apply helper; replace its review-pass “completed” projection with `applyImplementationPublished` and `applyImplementationMerged` adapter operations owned by the PR handoff helper.
   - Publication accepts configured Implementing or already-Ready-for-Review, moves to Ready for Review when needed, verifies the postcondition, and writes one marker-deduped comment containing the PR URL, reviewed candidate, and run reference.
   - Merge acknowledgement accepts configured Ready for Review or already-Done, moves to Done when needed, verifies the postcondition, and writes one distinct marker-deduped completion comment containing the recorded PR, reviewed candidate, and merge commit.
   - Preserve retry safety across partial failures: an explicit `--apply` retry must repair a missing status or comment without appending another lifecycle event or comment. If merge acknowledgement follows a durably published PR whose projection was incomplete, repair the publication projection first, then apply Done, so the final retry cannot strand or regress Linear. Never use Linear state/comments to decide a Factory transition.
   - Replace completion-oriented assertions in `test/factory-linear-implementation-apply.test.ts` and `test/factory-linear-adapter.test.ts` with publication/merge coverage for allowed and intervening statuses, fresh-state checks, false/thrown mutations, partial status/comment failure, ordered repair, and marker deduplication; update `test/factory-implementation-actions.test.ts` to prove review pass itself performs no ready/done projection.

5. **`factory inspect` expectations and operator-facing sources — document the human gate and new lifecycle owner.**
   - Extend `test/factory-inspect.test.ts` to show distinct `awaiting-pr-publication`, `awaiting-pr-merge` (with recorded PR/candidate), and merged `complete` states, with commandless publication/merge waits and byte-stable read-only inspection.
   - Update `README.md`, `docs/contributing/factory.md`, `docs/contributing/architecture.md`, `docs/contributing/script-command-surface.md`, and `docs/contributing/setup-manifest.md` for the two commands, event/state sequence, required Linear mappings, durable evidence, and the cutover from review-pass completion to human merge closure.
   - Update `skills/factory-operator/SKILL.md` command/config examples and procedure: passing review waits for explicit PR publication authority while Linear remains Implementing; once authorized, the executor may push/open and prepare the PR but must stop after `publish --apply`; opening/delivering never authorizes merge; the executor must not merge, and only after the human has merged that recorded PR may the operator fetch the merge commit and run `mark-pr-merged --apply`. State plainly that Harness commands own neither GitHub credentials nor push/open/merge operations.

- Live dogfood FER-78 through the new boundary: confirm review pass leaves Factory `awaiting-pr-publication` and Linear Implementing; after explicit publication authority, open and prepare the PR without merging; publish the recorded URL with `--apply`, retry once to prove one event/comment, confirm `factory inspect` reports `awaiting-pr-merge` and Linear Ready for Review, then stop and report the ready PR. Only after the human explicitly decides to merge and has merged that recorded PR, fetch the merge object, run `mark-pr-merged` with the same URL and merge SHA plus `--apply`, retry once, and confirm one merge event/completion comment, Factory `complete`, and Linear Done.

## Boundaries

- No GitHub adapter, credentials, polling, scheduler, hosted orchestrator, automatic merge, In Review station, generic publication abstraction, or compatibility path.
- Do not weaken candidate ancestry checks to tree/diff similarity, and do not accept GitHub or Linear observations as lifecycle inputs.
- STOP the dogfood at `awaiting-pr-merge` unless the human explicitly authorizes merging that recorded PR; if merge or authorization does not happen, leave Factory and Linear at their waiting states.
