# Publish reviewed Factory work through human-merged pull requests

## Goal

Close the Factory planning and implementation publication gaps with deterministic, manually authorized Harness handlers. A passing pull-request planning review waits for Harness to materialize and publish the reviewed plan; a passing implementation review waits for Harness to publish the exact promoted candidate. Publication records the PR and head, then projects Linear to Plan Needs Review or Ready for Review. Only explicit acknowledgement of a human merge may approve planning or complete implementation and move Linear to Ready to Implement or Done.

Git remains the committed-artifact source of truth and Factory events remain lifecycle truth. Planner and implementer sessions retain their current no-GitHub-mutation sandbox; the manual publication command, using local Git credentials and authenticated GitHub CLI, authorizes only its phase-specific push/open operation. No command merges, polls, infers a result from GitHub/Linear, or runs a provider/reviewer.

## Changes

1. **`lib/factory-phase-run.ts`, `lib/factory-planning-run-context.ts`, and `lib/factory-implementation-run-context.ts` — persist the Git inputs needed for retry-stable publication.**
   - For pull-request planning, snapshot the configured base ref, its exact starting commit, and the attached clean branch with the phase identity. Add the configured base ref to implementation identity while retaining its existing branch ref/base commit and candidate refs. Validate these values when reopening a run; publication must not silently rebase onto newer local or remote state.
   - Keep local item-file planning on its existing direct-materialization/`approved` path. PR-backed planning and every reviewed implementation use the publication boundary; implementation publication accepts the same item-file-or-Linear selector contract as `implementation run`, while `--apply` remains Linear-only.
   - Extend phase-run schema tests in `test/factory-action-kernel.test.ts` and Git precondition/reopen coverage in the existing planning and implementation run-context/action tests.

2. **New `lib/factory-pull-request-publisher.ts` plus phase preparation helpers — own the small shared Git/GitHub boundary.**
   - Expose one bounded publisher that receives an already-prepared base ref, head branch, exact head SHA, title, and body. Production execution uses non-interactive `git` plus `gh pr list --state all --head ... --json ...` / `gh pr create --base ... --head ... --title ... --body ...`; keep command execution injectable for tests rather than introducing a general GitHub adapter or SDK.
   - Before any mutation, identify and validate `origin`, the local head, any remote head, and any PR for the deterministic head branch. Push only when the remote ref is absent; reuse it only at the exact expected SHA; reject divergent refs, multiple/mismatched PRs, wrong bases, or a PR whose reported head SHA differs. Query before creation and query again after an ambiguous create failure so retries recover a push or PR that succeeded without a usable response.
   - Planning preparation builds `refs/heads/harness/factory/plan/<phaseRunId>` from the persisted base commit using the reviewed immutable plan. It deterministically upserts one canonical, sorted Active queue entry in `dev/plans/README.md`, rejects malformed/duplicate/conflicting plan or index state, creates one deterministic commit containing only the plan and index, and checks out/materializes that exact commit without carrying unrelated workspace changes.
   - Implementation preparation requires the persisted branch tip to equal the promoted candidate from the final passing review, then publishes that existing branch and SHA unchanged. No publication path accepts a pre-created URL, a plan path, arbitrary commit, or changed PR head.
   - Add `test/factory-pull-request-publisher.test.ts` with temporary local/bare Git repositories and a fake `gh` runner for missing/exact/divergent remote refs, existing/ambiguous/mismatched PRs, create-response loss, deterministic plan commit/index bytes, and exact implementation head enforcement.

3. **`lib/factory-lifecycle-events.ts` and `lib/factory-state-machine.ts` — add explicit publication and human-merge states.**
   - A passing PR-backed planning review reduces to `awaiting-plan-publication`; `plan_pr.opened` records the PR URL, exact published head, and reviewed plan and moves to `awaiting-plan-merge`; `plan_pr.merged` records the same URL and acknowledged merge commit and moves to `approved`. Require the head on the opened event and do not retain the old publication command contract.
   - A passing implementation review reduces to `awaiting-pr-publication`, not `complete`. Add strict, phase-run-scoped `implementation_pr.opened` `{ url, head }` and `implementation_pr.merged` `{ url, commit }` facts; opened is legal only for the final reviewed/promoted head and moves to `awaiting-pr-merge`, while merged is legal only for the recorded URL and moves to `complete`. Preserve PR URL, reviewed/published head, and merge commit in the read model.
   - Add commandless `plan-publication`, `pr-publication`, `plan-merge`, and `pr-merge` wait reactions. Cover schemas, legal/illegal transitions, immutable URL/head binding, rebuilds, and reactions in `test/factory-action-kernel.test.ts`; update review-pass and recovery assertions in `test/factory-planning-actions.test.ts` and `test/factory-implementation-actions.test.ts`.

4. **New `lib/factory-plan-publication.ts` and `lib/factory-implementation-publication.ts`, with thin registration in `bin/factory-planning-cli.ts` and `bin/factory-implementation-cli.ts` — publish and acknowledge without agent authority.**
   - `publishPlanPullRequest` and `publishImplementationPullRequest` resolve the active phase and final passing review, invoke only their phase preparation plus the shared publisher, then append deterministic opened events before optional Linear projection. Remove planning `publish --url/--plan`; the command now derives both. Add `implementation publish` with workspace/store and item-file/Linear selectors. Command invocation itself authorizes one push/find-or-create operation; `--apply` additionally authorizes its Linear projection.
   - Retry in this order: reuse an identical durable opened event when present; otherwise reconcile remote branch and PR, append the lifecycle fact, then project. Thus failure after push, PR creation, event append, or Linear mutation resumes without a second branch/PR/event/comment; changed URL/head/base/branch conflicts with durable truth and stops.
   - Keep `planning mark-plan-merged --url <url> --commit <sha>` and add `implementation mark-pr-merged --url <url> --commit <sha>`, both with the same selector/store options and optional `--apply`. Each requires the supplied URL to match its opened fact, resolves the supplied SHA as a local commit, and requires the recorded published head to be its ancestor before appending one deterministic merged event. Implementation closure therefore rejects an unrecorded/wrong PR, missing/non-commit/wrong-ancestry SHA, or absent publication without querying GitHub or merging anything.
   - Remove review-pass ownership of `applyImplementationCompleted` and its terminal repair path; publication and merge handlers become the only ready-PR and completion owners. Update `assertLiveImplementationStatus` for Implementing, Ready for Review, and Done waits without allowing tracker state to drive lifecycle transitions.
   - Add phase-handler tests in `test/factory-planning-cli.test.ts` and new `test/factory-implementation-publication.test.ts` using real Git ancestry plus fake publisher/Linear dependencies for exact candidate selection, repair descendants in an acknowledged merge, all rejection cases, external-side-effect ordering, identical retries, and projection-only repair. Extend `test/factory-implementation-cli.test.ts`, `test/cli.test.ts`, and `scripts/smoke-dist.ts` for the changed planning options and both implementation subcommands.

5. **`lib/schemas.ts`, `lib/config.ts`, root `harness.json`, and Linear adapter helpers — project publication and closure explicitly and idempotently.**
   - Require `factory.linear.statuses.readyForReview` and `done`; keep only `canceled` and `duplicate` optional. Add the root Ready for Review mapping, include all implementation stations in distinctness/status-map validation, and update the targeted missing-mapping diagnostic and config/fixture coverage in `test/config.test.ts` and `test/factory-linear-test-helpers.ts`.
   - Move ready-PR and merge projection behavior into new `lib/factory-linear-implementation-handoff.ts`, exposed through `lib/factory-linear-adapter.ts`; retain start, attention, and failure in `lib/factory-linear-implementation-apply.ts`. Publication accepts Implementing/already-Ready-for-Review, sets and verifies Ready for Review, and writes one marker-deduplicated comment containing PR URL, reviewed head, and run. Merge accepts Ready for Review/already-Done, sets and verifies Done, and writes a distinct completion comment containing PR, reviewed head, and merge commit.
   - If merge acknowledgement follows a durable publication whose Linear projection failed, repair the publication projection before applying Done; never regress an already-Done issue. Lifecycle append remains before projection, and retry must repair a missing status/comment without duplicating either lifecycle fact. Replace completion-pass expectations in `test/factory-linear-implementation-apply.test.ts` and `test/factory-linear-adapter.test.ts` with publication/merge status allowlists, mutation/postcondition failures, ordered partial-failure recovery, and marker deduplication.

6. **`factory inspect` contracts and operator-facing docs — expose the two human gates and new mutation owner.**
   - Extend `test/factory-inspect.test.ts` for awaiting plan publication/merge and implementation publication/merge/complete, including recorded PR/head/merge fields, commandless reactions, and byte-stable read-only output.
   - Update `README.md`, `docs/contributing/factory.md`, `docs/contributing/architecture.md`, `docs/contributing/script-command-surface.md`, and `docs/contributing/setup-manifest.md` with the handler/event sequence, Git/GitHub/Factory authority split, changed command flags, required Linear mappings, GitHub CLI plus Git remote authentication, retry behavior, and external mutations performed by publication commands. Remove current statements that Factory commands never mutate GitHub.
   - Update `skills/factory-operator/SKILL.md` to keep planner/implementer sessions credential-free, invoke publication only after explicit phase authorization, stop and report the ready PR, and treat opening/delivering a PR as no merge authority. Only after a human has merged the recorded PR should the operator fetch the merge commit and run the corresponding mark command.

## Verify

- Run focused Vitest files for the shared publisher, lifecycle/action recovery, phase publication handlers, Linear handoff/config, inspect, CLI, docs, and packaged skill contracts; then run canonical `pnpm check`.
- Dogfood FER-78 with authenticated Git/`gh`: confirm implementation review pass leaves Factory `awaiting-pr-publication` and Linear Implementing; run `implementation publish --apply`, retry once, and confirm one remote branch, PR, opened event, and Linear comment plus `awaiting-pr-merge`/Ready for Review inspection. Stop and report the PR. Only after explicit human merge, fetch the merge commit, run `mark-pr-merged` with that PR URL/SHA and `--apply`, retry once, and confirm one merged event/completion comment, Factory `complete`, and Linear Done.

## Boundaries

- No automatic merge, GitHub polling/merge adapter, PR reviewer agent, Inngest integration, scheduler, hosted workspace/store, generic publication framework, or provider/reviewer invocation from publication/merge commands.
- Do not accept diff/tree similarity, Linear state, or GitHub merge observations as lifecycle evidence; publication uses the exact reviewed head and closure uses explicit human-supplied URL/SHA plus local Git ancestry.
- STOP dogfood at `awaiting-pr-merge` unless the human explicitly authorizes that recorded PR's merge. If it is not merged, leave Factory and Linear waiting.
