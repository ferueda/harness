# Deliver one Factory operation through Inngest

## Goal

Add the first Inngest adapter around `runHostedFactoryOperation`: one versioned, identifier-only event must deliver and retry one exact Factory operation while Factory JSONL, action identity, saved-result recovery, and reaction selection remain authoritative. The adapter is complete when an authenticated durable receipt (including one derived from `factory.action.failed`) completes its operation step, only a receipt with `next` emits another durable event, host failure before a receipt remains retryable by Inngest, and duplicate or stale delivery cannot execute a different reaction inside the same step.

Use app ID `harness-factory`, function ID `execute-factory-operation-v1`, and event `harness/factory.operation.requested` with event version `"1"`. Limit active steps to one per work item and one globally as scheduling controls only. Keep the initial deployment model to one persistent host; this change adds no ingress or production deployment entrypoint.

## Changes

1. `package.json` and `pnpm-lock.yaml` — add exact, non-floating dependencies for the verified v4 surface: runtime `inngest@4.13.0`, and development `@inngest/test@1.0.0` plus `inngest-cli@1.37.0`. These were the stable registry versions on 2026-07-16; recheck before installing and use only a mechanical newer stable-v4/tool pin. STOP if the supported API has moved to v5 or no longer provides typed versioned events, two function concurrency limits, `step.run`, `step.sendEvent`, or Connect. Do not commit any `latest` command or range.

2. Add `lib/factory-inngest-adapter.ts` with the fixed identifiers, `FactoryOperationRequestedEvent`, and `createFactoryInngestAdapter`. The factory accepts a caller-owned Inngest client and a trusted `HostedFactoryRuntime`, rejects a client whose ID is not `harness-factory`, closes over both, and returns the single registered function; an injected runner is allowed only as a focused test seam. The runtime remains the sole owner of Factory-store paths, Grove configuration, provider construction, and other host controls: never serialize it into or reconstruct it from event data, and leave production runtime/client construction and deployment wiring out of scope. Define the event with v4 `eventType()` using `FactoryOperationRequestSchema` as its strict `data` schema and version `"1"`; do not add paths, credentials, work-item contents, Git/Grove details, or lifecycle state to the event. The factory must create one `execute-factory-operation-v1` function with `concurrency: [{ key: "event.data.workItemKey", limit: 1 }, { limit: 1 }]` and the single trigger above; these limits queue work but are not idempotency or locking authority.

   Inside the function, place exactly one call to `runHostedFactoryOperation` in stable step `run-factory-operation-v1`. Use the lesser of the injected runtime budget and 30 minutes (the existing Harness action default and below the currently documented two-hour Inngest step maximum), combine a timeout signal for that budget with any host-supplied runtime signal, and pass the combined `AbortSignal` and bounded `maxRuntimeMs` into the runner. This is the active-step cancellation path: current Inngest cancellation does not interrupt an already executing step, so do not claim that dashboard/event cancellation can stop Factory mid-action.

   Treat every returned `FactoryOperationReceipt` as a successful step result, including an executed or recovered durable `factory.action.failed` result. If the deadline signal causes an existing Factory action to persist an aborted failure and the runner returns its authenticated receipt, succeed the Inngest step and use only that receipt's `next`, if present; do not add a lower-level cancellation state or turn the durable result back into a host retry. If the runner rejects before any receipt exists, let that error throw so Inngest retries the identical event, while the runner owns saved-result recovery on replay. Do not catch an error and invoke another operation in the same step. After the operation step, call durable `step.sendEvent("send-next-factory-operation-v1", FactoryOperationRequestedEvent.create(receipt.next))` only when `next` exists, then return the original identifier-only receipt. Never loop, follow `next` directly, use `inngest.send()`, or emit for stale/waiting receipts; human, phase-command, publication, merge, complete, failed, and stale waits therefore stop at this function run.

3. Add `lib/factory-inngest-adapter.test.ts` using `@inngest/test` with Vitest and the existing isolated hosted-operation fixtures. Lock the fixed app/function/event/step IDs, event version and strict request data, the two concurrency controls, the 30-minute runtime cap/signal forwarding, and identifier-only function/step/send-event results. At the function boundary, prove:

   - a current request executes the real hosted runner once and a durable retryable `factory.action.failed` receipt succeeds while emitting exactly its newly derived next-operation event;
   - two deliveries of the same request yield executed then recovered receipts through Factory action identity, call the provider once, and never advance to the hinted reaction inside either operation step;
   - stale delivery and representative human, publication/merge, complete, failed, and stale waits return without `step.sendEvent`;
   - a Grove/host failure before action-result persistence rejects the step without an event, while provider or deadline abort persisted and authenticated by Factory resolves successfully without an Inngest host retry; and
   - a forced failure of `send-next-factory-operation-v1` after Factory has persisted the receipt and derived `next` leaves `run-factory-operation-v1` checkpointed; replay sends the same derived operation without rerunning the provider, changing Factory evidence, or advancing to another reaction inside the operation step.

   Keep the existing `lib/factory-hosted-operation*.test.ts` recovery and identity matrix as the lower seam; do not duplicate its tamper, Git, or Grove divergence cases in the Inngest test.

4. `scripts/smoke-factory-grove.ts` — cut the existing isolated hosted execution/replay journey over from direct runner calls to the real Inngest adapter. Allocate separate loopback HTTP and Connect-gateway ports, strip every inherited `INNGEST_*` value from the smoke and Dev Server child environments, and start the pinned local Dev Server with those exact `--host`, `--port`, and `--connect-gateway-port` values plus `--no-discovery` and `--no-poll`, with no `--sdk-url`/`-u`. Pinned 1.37.0 CLI help/source confirms that these flags disable localhost discovery and app polling and that an omitted SDK URL leaves the serve-endpoint list empty; registration must come only from the explicitly configured Connect worker. After bounded HTTP readiness succeeds, set only test-local client routing (`INNGEST_DEV` and `INNGEST_BASE_URL` to the allocated HTTP URL plus a disposable `INNGEST_EVENT_KEY` if required by the verified v4 send path), then construct the caller-owned client and one in-process Connect worker. Pass `connect()` the exact `ws://127.0.0.1:<gateway-port>/v0/connect` gateway URL, `maxWorkerConcurrency: 1`, a stable smoke instance ID, and disabled execution isolation for the injected disposable runtime; never fall back to an inherited Cloud base, signing/API key, environment, or gateway.

   Send the prepared versioned event through that client and poll the pinned CLI/API with explicit `--api-host <allocated-http-url>` targeting. Require local run metadata/diagnostics to identify app `harness-factory` and function `execute-factory-operation-v1`; assert the first event ID resolves on that Dev Server to an `executed` output, then release the terminal Grove lease and require the redelivery event ID on the same server to resolve to `recovered`. Preserve the current assertions that provider/setup run once, the lease is not recreated, and Factory lifecycle/action evidence does not change on replay. Always close the Connect worker and Dev Server process; retain the isolated fixture, allocated URLs, registration/run metadata, and bounded child output on failure, and clean the fixture on success. Do not add a reusable HTTP server, production worker command, live credentials, or external service dependency.

5. `docs/contributing/architecture.md`, `docs/contributing/factory.md`, `docs/contributing/testing.md`, `docs/contributing/script-command-surface.md`, and `test/docs-contracts.test.ts` — replace the statement that no scheduler ships with the narrow current boundary: Inngest delivers/retries one identifier-only operation and may emit one hinted operation, while Factory still owns transitions, retries, persistence, publication, and correctness. Record the injected trusted-runtime boundary, fixed event/function IDs, scheduling-only concurrency, 30-minute signal, successful persisted-abort/durable-failure semantics, pre-receipt host retry behavior, single-host limit, and the local Dev Server + Connect smoke. Add focused doc sentinels for the adapter owner and smoke without describing webhook ingestion, a production deployment, or a long-running Inngest workflow.

## Boundaries

- Do not add Linear/GitHub webhook ingestion, `waitForEvent`, a phase/review loop, routing, publication, merge acknowledgement, Grove release, or any second Factory state/retry policy to Inngest.
- Do not use Inngest event IDs, idempotency, concurrency, function attempts, or step state as Factory correctness authority; duplicate events must still converge through the requested Factory action identity and durable result.
- Do not claim multi-worker correctness or add horizontal-scaling controls. Connect is used only by the disposable local smoke; production connection and secret wiring require a later work item.
